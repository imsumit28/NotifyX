NotifyX · Integration Guide

Add real-time notifications
to any project

NotifyX runs as a standalone microservice alongside your app. Your backend sends one HTTP request; your frontend connects via Socket.io. Everything else — queuing, retries, batching, offline sync — is handled automatically.

Get an API key

Anyone who wants to integrate opens the dashboard, signs up, and generates their own key from the API Keys tab — no admin involvement needed.

Open the dashboard Go to localhost:8080/dashboard.html (or your deployed URL)
Sign up Create an account with any user ID and password — takes 10 seconds
API Keys tab → Generate Key Give it a name (e.g. my-blog), click Generate — the full key is shown once
Copy and store it Save as NOTIFYX_API_KEY in your project's .env — it won't be shown again

The key is hashed (SHA-256) before storage — the server never holds the raw value. If lost, generate a new one and revoke the old one from the API Keys tab.

Or generate one directly with curl if you prefer:

bash 
curl -X POST http://localhost:3000/api/keys \
  -H "x-admin-secret: notifyx-demo" \
  -H "Content-Type: application/json" \
  -d '{"appName":"my-blog"}'

# { "key": "nx_a1b2c3...", "note": "Save this — shown once only" }

Send notifications

One POST request from your backend. NotifyX queues the job, processes it, and delivers to the recipient's browser in under 100ms.

javascript · node.js 
async function notify({ recipientId, senderId, type, payload }) {
  const res = await fetch(`${process.env.NOTIFYX_URL}/api/notify`, {
    method: 'POST',
    headers: {
      'Authorization':  `ApiKey ${process.env.NOTIFYX_API_KEY}`,
      'Content-Type':   'application/json',
      'Idempotency-Key': crypto.randomUUID(),
    },
    body: JSON.stringify({ recipientId, senderId, type, payload }),
  });
  if (!res.ok) throw new Error('notify failed');
  return res.json(); // { status: 'queued', jobId: '...' }
}

// comment on a post
await notify({
  recipientId: 'user_alice',
  senderId:    'user_bob',
  type:        'comment',   // like | comment | follow | mention
  payload:     { message: 'Nice work!', link: '/posts/123' },
});
python 
import requests, uuid, os

def notify(recipient_id, sender_id, notif_type, payload):
    r = requests.post(
        f"{os.environ['NOTIFYX_URL']}/api/notify",
        json={
            "recipientId":    recipient_id,
            "senderId":       sender_id,
            "type":           notif_type,
            "payload":        payload,
            "idempotencyKey": str(uuid.uuid4()),
        },
        headers={"Authorization": f"ApiKey {os.environ['NOTIFYX_API_KEY']}"},
        timeout=5,
    )
    r.raise_for_status()
    return r.json()

Real-time delivery (Socket.io)

Connect with a JWT token. Notifications arrive in under 50ms when the user is online. If they're offline, they're synced automatically on the next connection.

javascript · browser 
// 1. get a token (signup once, login after)
const { token } = await fetch('http://localhost:3000/api/auth/login', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({ userId: 'user_alice', password: '…' }),
}).then(r => r.json());

// 2. connect
const socket = io('http://localhost:3000', {
  auth: { token },
  reconnection: true,
});

// 3. listen
socket.on('notification', (n) => {
  // n = { _id, recipientId, senderId, type, payload, status, createdAt }
  showToast(n.payload.message);
  incrementBadge();
});

socket.on('connect',       () => console.log('connected — missed notifs will sync'));
socket.on('connect_error', (e) => console.error('auth error:', e.message));
react hook · drop-in 
import { useEffect, useState } from 'react';
import { io } from 'socket.io-client';

export function useNotifyX(token) {
  const [notifications, setNotifications] = useState([]);
  const [unread, setUnread]               = useState(0);

  useEffect(() => {
    if (!token) return;
    const s = io('http://localhost:3000', { auth: { token } });
    s.on('notification', n => {
      setNotifications(prev => [n, ...prev]);
      setUnread(c => c + 1);
    });
    return () => s.disconnect();
  }, [token]);

  return { notifications, unread };
}

Metrics & charts

GET /api/metrics returns live queue stats and delivery rates. Poll it every few seconds to drive a chart.

javascript 
const m = await fetch('/api/metrics', {
  headers: { 'Authorization': `Bearer ${token}` },
}).then(r => r.json());

/*
  m.jobs:     { waiting, active, completed, failed, delayed }
  m.delivery: { total, success, failed, successRate, failureRate }
  m.workers:  { active: ['worker-1'], count: 1 }
*/

// live chart — refresh every 3s
setInterval(async () => {
  const data = await fetchMetrics(token);
  chart.update({
    successRate: parseFloat(data.delivery.successRate),
    queued:      data.jobs.waiting + data.jobs.active,
  });
}, 3000);

The built-in dashboard already has live charts. Open dashboard.html → Metrics tab to see delivery rates, queue depth, and active workers updating in real time.

User preferences

Users can mute notification types, set quiet hours, or disable all delivery. The worker checks preferences before delivering — a muted type is silently dropped.

javascript 
// read
const prefs = await fetch('/api/users/preferences', {
  headers: { 'Authorization': `Bearer ${token}` },
}).then(r => r.json());
// { inApp: true, mutedTypes: [], quietHours: { enabled: false, startHour: 22, endHour: 8 } }

// update — mute likes, enable quiet hours 11pm–7am
await fetch('/api/users/preferences', {
  method: 'PUT',
  headers: {
    'Authorization': `Bearer ${token}`,
    'Content-Type':  'application/json',
  },
  body: JSON.stringify({
    mutedTypes: ['like'],
    quietHours: { enabled: true, startHour: 23, endHour: 7 },
  }),
});

Delivery flow

What happens between POST /api/notify and the notification appearing in the browser.

Auth & validateJWT or ApiKey check → Joi schema → Redis SETNX idempotency (24h TTL)
Enqueue to BullMQJob stored in Redis. 202 returned to caller immediately. Priority 1–10.
Worker picks up job2nd idempotency check in MongoDB. Load user prefs (Redis cache → DB fallback).
Preference checksSkip if inApp off, type muted, or quiet hours active.
Save to MongoDBNotification document created. delivered: true if user online, false if not.
Redis Pub/Sub publishWorker publishes to user:{recipientId}. API server's subscriber picks it up.
Socket.io emitIf online: delivered live in <50ms. If offline: stored as delivered:false, pushed on next connect.

FAQ

How do I prevent duplicate notifications?

Pass a unique idempotencyKey with each request. NotifyX uses two layers: Redis SETNX at the API boundary (24h TTL) and a MongoDB sparse unique index in the worker. Sending the same key twice returns 409 Conflict.

What notification types are supported?

like, comment, follow, mention. To add more, update NOTIFICATION_TYPES in shared/constants.js and the Joi enum in server/src/routes/notify.js.

What is automatic batching?

High-volume types (likes, follows) within a 30-second window are grouped into one notification. Five people liking the same post becomes "5 people liked your post." Configured via BATCH_TYPES in shared/constants.js.

What happens if the worker crashes mid-job?

BullMQ holds a lock on active jobs. If the worker dies, the lock expires in 30 seconds and another worker instance picks it up. After 5 failed attempts the job moves to the dead-letter queue where you can inspect and retry it.

Can I run multiple workers?

Yes — start multiple worker processes pointing at the same queue. They compete for jobs automatically. Each registers a heartbeat key in Redis; the Metrics API lists them under workers.active.

If load increases, does it cost the server owner?

Yes — all traffic runs through one shared Redis and MongoDB instance. For development and demos this is fine. For serious production use, each integration should deploy their own copy of the stack (15-minute setup, all free tiers).
